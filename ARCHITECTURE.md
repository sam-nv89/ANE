# Архитектура Adaptive Nutrition Engine

Проект построен как масштабируемое SPA на React, ориентированное на сложную бизнес-логику работы с данными пользователя: расчеты нормативов, фильтрация рецептов и умное построение 7-дневного плана по множеству критериев.

## 📂 Структура проекта

```text
src/
├── components/          # Reusable UI компоненты (карточки, модалки, кнопки)
├── data/
│   └── recipes.json     # Статическая (JSON) база рецептов
├── lib/
│   ├── nutrition/       # Расчет КБЖУ, формулы, фильтры аллергенов
│   └── planner/         # Ядро генерации сценариев: timeFilter, monotonyIndex, generator
├── pages/               # Компоненты-страницы (Pages)
│   ├── AppShell/        # Root Layout приложения
│   ├── Dashboard/       # Главный дашборд
│   ├── MealDetail/      # Карточка рецепта
│   ├── Onboarding/      # Опросник
│   ├── Profile/         # Редактирование профиля
│   ├── Progress/        # Трекинг
│   └── ShoppingList/    # Список покупок
├── store/               # Zustand стейты (пользователь, план, покупки)
├── App.jsx              # Конфигурация роутеров (React Router v7)
└── index.css            # Дизайн-система (CSS variables, utilities)
```

## 🧠 Ядро генерации (Planner)

Главная архитектурная ценность приложения заключается в алгоритме построения рациона, который решается в `src/lib/planner/generator.js`.

### Пайплайн генерации `generateWeeklyPlan`:
1. **Расчет матрицы КБЖУ**: Используется `getCalorieDistribution` из `src/lib/nutrition/distribution.js`. На базе TDEE высчитываются плановые показатели, которые динамически нормализуются в зависимости от количества приемов пищи (от 3 до 7) и настроек профиля (`heavy-breakfast`, `light-dinner` и др.).
2. **Фильтрация базы**: Все рецепты фильтруются через аллергены и диетические предпочтения пользователя (`filterSafeRecipes`).
3. **Time-based фильтрация**: Рецепты фильтруются по наличию доступного времени у пользователя `filterByTimeWindow`.
4. **Матричный подбор (Fallback Strategy)**: Выбор блюда на конкретный слот осуществляется по 4 линиям деградации:
   - **Lvl 1:** Попадание в ±30% от целевых ккал слота + прохождение чека на монотонность.
   - **Lvl 2:** Попадание в ±30% от целевых ккал, монотонность игнорируется.
   - **Lvl 3:** Попадание в ±45% от целевых ккал.
   - **Lvl 4:** Аварийный (`closest`) — любой доступный рецепт из пула, ближайший по ккал (во избежание возврата пустых слотов).
5. **Адаптация времени (Self-healing)**: Если time-фильтр слишком жесток и оставляет менее 3 рецептов, алгоритм автоматически локально расширяет окно (×1.5).

## 🗄 Управление состоянием (Zustand)

Изоляция состояния от UI позволяет производить пересчеты незаметно для компонентов.

- **`useUserStore`**: Хранит профиль пользователя (пол, возраст, рост, вес) и вычисленную биометрию (BMR, TDEE, Macros). Подключен к `localStorage`, чтобы данные не пропадали.
- **`usePlanStore`**: Хранит текущий сгенерированный 7-дневный план и состояние загрузки (`isLoading`).
- **`useShoppingStore`**: Управляет отметками покупок в чеклисте.
- **`useProgressStore`**: История изменений веса, статистика соблюдения меню.

## 🎨 Design System

Логика стилей изолирована на уровне `index.css`. Используется тёмная высококонтрастная тема, основанная на CSS variables.
- **Skeleton System**: Реализована система анимированных заглушек (`src/components/Common/Skeleton.jsx`) с эффектом `shimmer` для визуализации процесса загрузки и генерации данных (предотвращение CLS — Cumulative Layout Shift).
- **Динамическое масштабирование**: Граммовки рецептов и БЖУ рассчитываются на лету через `planMultiplier` (подстройка под калораж) и `portionsMultiplier` (выбор количества порций на экране `MealDetail`).

Анимации вынесены в слой `Framer Motion` внутри конкретных React-компонентов.
